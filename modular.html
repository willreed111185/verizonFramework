Modular

It is important to “Think like a Developer” when approaching Robotic Process Automation (RPA). Nowhere is this more important than the design and implementation concepts of modularity.

Why Be Modular

Modularity is a concept, it's not finite. Think of it like organization, it's all relative really. However, it lends an ability to be a more effective developer, to have more organized code, to plan accordingly, to have small victories, to fight small battles, and to reuse what you've created.

Lets Make Coffee

When you think about all of the tasks to complete a process, the list can be daunting. Let’s think about the process of getting ready for work. For some, this means making taking a shower, making coffee, walking the dog, getting the kids ready and maybe even packing lunches. If we were to write this all out, line by line, it would probably be a usable process, which will work for one person. However, let’s say we write down directions for making coffee. We also write down directions for packing a lunch, showering, and everything we mentioned previously.

Now, we have a list of independent functional tasks. They may be dependent on other things, and that’s fun. However, you have directions on how to make coffee. You can give them to your spouse, and now they can make coffee. Each of our children can now execute a sandwich making procedure and pack their own lunches. You can pass this process off to a coworker, and now just as simply as giving them your grandmothers’ cheesecake recipe they can function with your morning procedure.

This is the same thing for a machine.

Reduce, Reuse, Recycle

I'm sure we all heard this in elementary school regarding waste. This applies to coding as well. We want to reduce the amount of code we write, we want to reuse things which we have previously built, and we want to recycle processes from one project in another.

If you need to log into a system and check a status of something, that can be treated as a task and scripted as a task. That task can be called time and time again to do the same thing and it can be called by many different machines to do the same thing. It’s important to consider the ability to independently operate given a certain set of inputs, and to use those as variables passed to the subtask (also referred to as an argument). If you’re logging into a site, make sure the subscript has the URL for the site, and the username/password combination handed to it from the main task. This way, many different credentials can be used and a change to the URL can be handled (think XML).

Now, we have smaller and more manageable tasks and we have an ability to reuse their functionality.
