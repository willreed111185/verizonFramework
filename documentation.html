  • Download the framework from the Repository at "My Tasks\NewAutomationProject_V4".

What It Does

This first page is intended to walk you through the process to make sure you have all of the prerequisites for the framework.

The framework uses a group XML, this tells the Main Framework Task (Main) what CyberArk Vault to import, what share drive you will be using, as well as which group you are a part of.
This file is the first thing which needs to be downloaded from the repo.

The framework then needs the Main ATMX file to be downloaded from the repo. This is what the developer will launch and what till build all of the proceeding elements.

When Main is ran, it will ask you for a project name. Let’s use “Blue $Company!” as an example. The script will first strip any special characters and white space, then it will add “_$System(USERNAME)$_$Day$_$Month$_$Year$_V1” to the end of it. You’d be left with “BlueCompany_reedwi4_09-19-2017_V1.

Main will then create the folder structure on the share drive and the local machine, copy all of the template tasks, then write the XML file. Below is a tree of the folder structure and files.

MainTask

The first section covers priming the status variables (used for flow control), initializing the Timestamp, resolving the Task Name (programatically), then running the XML and CyberArk subtask.A few more tests and variable initializations complete the first section of this code.

The next section is the Environmental Housekeeping section. This section is responsible for assuring that all of the necessary background application status's are correct and that all file/folders that may need to be removed are done so.

The next section of the code, and is responsible for copying all of the inputs from the share drive (Input Folder and Log Files), zips the Input Folder into the Archive Folder on the share drive, and deletes the Input Folder on the share drive (so that it's empty for the next iteration). This deletion feature can be turned off if it isn't needed (don't forget to delete the step that recreates the folder as well).

This section of the code is responsible for calling a subtask. The If/End If is what handled the flow-control, and the logging is handled as well. This section can be replicated to handle many subtasks by simply copy/pasting the block of code.

This is an example of two subtasks being called. The only change is the URL for the subtask you are utilizing (You can rename the task from "1.0 Subtask" to something that better fits your task).

This section of the code handled the Business Unit Logging. The Framework determines whether the task and subtask completed successfully, and then notes the XLSX file accordingly.

This is the final section of code in the Main Task. This section handles the closing operations of the script. The files in the Working directory are zipped and copied to the sharedrive in the Output folder (zipping isn't a requirement but recommended due to ease and capacity). The Business Unit log file on the share drive is deleted before being replaced with the new log file Some final notes are logged in the COE file and the Main Task is completed.

XML and CyberArk

This is the first section of the XML/Cyberark Subtask. This is responsible for opening the XML file (in the background) and setting the individual variables according to the nodes.

The IF/ELSE section is designed allow the RPA to scale. it does this by choosing the file with the system(USERNAME) in the name by default. If you have one without the username in it, it will use that as a backup.

The second section of this subtask is used to call upon CyberArk (when not in "Run" mode), or to prompt for a password. This can be used and replicated to handle multiple passwords. However, only one mode can be selected at a time. The password in either case will be assigned to vPassword, but this can be changed to suit multiple passwords.

This is what the XML file looks like. Highlighted is the opening and closing "tags" (notice the slash in the closing tag). When you add a new element/variable to the XML file, this is the format which needs to be used. The line starting with containing the green text is a comment, and that formatting can be replicated to create a new comment if you choose to do so. The "Project-Name" opening and closing tags can be renamed to your project name, but this isn't absolutely necessary (it is recommended however).

Environmental Housekeeping

This short subtask calls on a .cmd file to be executed. This CMD file is exclusive to the project and it can be augmented to suit the individual needs of the task.

This is what the killTask looks like, feel free to change this as you need. It can be opened and edited in Notepad++ which should be available on all of the VMs.

Subtask

This is the smallest and likely the most important subtask. It's where the developer will be doing their work. As noted in the comments (please use lots of comments throughout your work), between the error handling brackets is where the process will begin and end for a particular subtask. Your variables from your XML will be passed to Main. Main will pass them to the subtasks, and here you will use them to do "things". Essentially, the framework is purposed to provide you with all of the structure, security, and usability features so that you can start it up and get to writing ASAP.
Enjoy!

Logging

There is a built in feature to the framework for logging your process in two ways, as a text document and as a spreadsheet. There is a slightly different process for a successful or unsuccessful process, and this can be changed by the developer to suit the needs of the particular process or reporting requirements.

This is a glimpse of the spreadsheet log template. It is designed to handle this column layout, however please feel free to augment this and expand it to handle as much as you'd like to report for the process you're developing.

This is a snapshot of a COE log file which has been used for a script in development. This offers a glimpse into the format and capability. Styling and formatting is important so that iterations and important information stand out. Please feel free to add styling to suit your needs, there is a lack of styling in this particular demo.
